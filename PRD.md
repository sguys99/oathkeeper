## 1. Executive Summary

OathKeeper는 B2B AI 개발 서비스 회사에서 세일즈 담당이 가져오는 고객사 개발 요청 사업 Deal의 수행 여부를 체계적으로 분석하고, **Go / No-Go / 조건부** 판단을 지원하는 AI 에이전트 서비스이다.

현재 구두 보고 기반의 비구조적 의사결정 프로세스를 데이터 기반의 일관된 평가 체계로 전환하여, 의사결정 속도 향상, 리스크 사전 식별, 구성원 간 판단 기준의 일관성을 확보하는 것이 목표이다.

## 2. Problem Statement

### 2.1 현재 의사결정 프로세스 (As-Is)

| 단계 | 현재 프로세스 | 문제점 |
|:---:|---|---|
| 1 | 세일즈가 고객사와 협의 | 협의 내용이 비구조적으로 기록됨 |
| 2 | 구두로 경영진/기술팀에 보고 | 정보 누락 및 전달 편향 발생 |
| 3 | 회의에서 토론 후 판단 | 판단 기준이 매번 달라지고, 의견 충돌 발생 |
| 4 | Deal 수행 중 이슈 발생 | 사전 리스크 분석 부재로 추가 리소스 투입 |

### 2.2 핵심 Pain Points

1. **의사결정 기준의 비일관성:** 판단 기준이 사람마다, 시점마다 달라져 구성원 간 의견 차이 발생
2. **리스크 분석 부재:** Deal 수락 후 수행 중 예상치 못한 이슈가 발생하여 추가 리소스 투입 필요
3. **의사결정 소요 시간 과다:** 비구조적 보고와 회의 기반 프로세스로 인해 의사결정에 많은 시간이 소요

## 5. Core Features

### 5.1 Feature 1: Deal 정보 입력 및 구조화

세일즈가 다양한 형태로 입력한 Deal 정보를 AI가 파싱하여 구조화된 리포트로 자동 정리한다.

**입력 방식 (MVP)**

- **Notion 구조화 폼:** 고객사명, 프로젝트 개요, 예상 금액, 납기, 기술 요구사항 등 필수 필드 입력
- **자유 텍스트 입력:** 회의록, 메모 등 비구조화 텍스트를 LLM이 파싱하여 필수 필드 추출
- **문서 업로드:** Word/PDF 형태의 회의록, RFP, 제안요청서 등을 업로드하면 핵심 정보 자동 추출

**구조화 출력 항목**

| 필드명 | 설명 | 입력 유형 |
|---|---|---|
| 고객사 정보 | 회사명, 규모, 산업군, 담당자 | 텍스트 / 선택 |
| 프로젝트 개요 | 요청 배경, 목적, 기대 효과 | 텍스트 (자유 입력) |
| 기술 요구사항 | 필요 기술 스택, 모델, 데이터 환경 | 텍스트 / 문서 업로드 |
| 예상 금액 및 납기 | 계약 금액, 요구 납기, 결제 조건 | 숫자 / 날짜 선택 |
| 특이사항 | 고객 특별 요구, 제약 조건, 보안 요건 등 | 텍스트 (자유 입력) |

---

### 5.2 Feature 2: 자동 스코어링 및 Go/No-Go 판단

7개 표준 평가 기준에 따라 각 Deal을 자동 스코어링하고, 종합 판단을 제시한다.

**평가 기준 및 가중치**

| # | 평가 기준 | 가중치 | 평가 항목 (예시) |
|:---:|---|:---:|---|
| 1 | 기술 적합성 | 20% | 자사 기술 스택으로 구현 가능한가? 추가 기술 학습 필요성은? |
| 2 | 리소스 가용성 | 15% | 현재 투입 가능한 인력/장비가 있는가? 타 프로젝트와 충돌은? |
| 3 | 수익성 | 20% | 예상 매출 대비 투입 비용이 적절한가? 목표 마진을 충족하는가? |
| 4 | 납기 리스크 | 15% | 고객 요구 일정이 현실적인가? 버퍼 기간이 충분한가? |
| 5 | 요구사항 명확성 | 10% | 고객의 요구사항이 구체적인가? 범위 변경 리스크는? |
| 6 | 전략적 가치 | 10% | 레퍼런스 확보, 신규 산업 진출, 기술 축적 등 장기적 가치 |
| 7 | 고객 리스크 | 10% | 고객사 지불 능력, 의사결정 구조, 과거 협업 이력 |

> **참고:** 가중치는 초기 기본값이며, 관리자가 설정 화면에서 조정 가능하도록 구현한다.

**종합 판단 로직**

| 판단 | 조건 | 설명 |
|---|---|---|
| ✅ **Go** | 종합 70점 이상 | 수행 권고. 주요 리스크 요인 함께 제시 |
| ⚠️ **조건부 Go** | 종합 40~69점 | 조건 충족 시 수행. 해소 필요 항목 명시 |
| ❌ **No-Go** | 종합 40점 미만 | 수행 비권고. 주요 사유 및 대안 제시 |
| ℹ️ **보류** | 필수 평가 항목 중 미입력 존재 | 판단 보류. 추가 확인이 필요한 항목을 구체적으로 안내 |

---

### 5.3 Feature 3: 리스크 요인 분석 및 경고

각 Deal의 잠재적 리스크 요인을 카테고리별로 식별하고, 심각도에 따른 경고를 제공한다.

**리스크 카테고리**

- **기술 리스크:** 미경험 기술, 높은 구현 복잡도, 데이터 품질 이슈 등
- **일정 리스크:** 촉박한 납기, 불명확한 마일스톤, 병렬 작업 불가 등
- **재무 리스크:** 낮은 마진, 불명확한 결제 조건, 추가 비용 발생 가능성 등
- **고객 리스크:** 의사결정 지연 가능성, 요구사항 변경 빈도, 지불 안정성 등
- **범위 리스크:** Scope creep 가능성, 모호한 성공 기준, 암묵적 추가 요구 등

**경고 레벨**

각 리스크 항목을 🔴 **높음** (미해결 시 프로젝트 실패 가능성) / 🟡 **중간** (추가 비용 발생 가능성) / 🟢 **낮음** (관리 가능한 수준)으로 분류하여 표시한다.

---

### 5.4 Feature 4: 과거 유사 프로젝트 검색 및 비교

과거 수행한 프로젝트 이력을 기반으로 유사한 사례를 검색하고, 해당 사례의 결과와 교훈을 함께 제공한다.

**기능 상세**

- **유사성 기준:** 산업군, 기술 스택, 프로젝트 규모, 고객 유형 등을 기반으로 유사도 점수 산출
- **비교 정보:** 유사 프로젝트의 실제 투입 리소스, 발생 이슈, 최종 결과(성공/실패/초과) 제공
- **Lessons Learned:** 유사 프로젝트에서 발생한 이슈와 그 해결 방법을 참고 정보로 제공

**데이터 수집 전략 (MVP)**

현재 과거 프로젝트 데이터가 여러 곳에 산재해 있으므로, 단계적 데이터 구축 전략을 적용한다:

1. **Phase 1 (MVP):** 관리자가 과거 프로젝트 정보를 구조화된 폼으로 수동 입력 (최소 10~20건 목표)
2. **Phase 2:** 산재된 문서(메일, 문서 등)를 LLM이 자동 파싱하여 DB 구축
3. **Phase 3:** HubSpot CRM 연동으로 자동 데이터 수집 및 업데이트

## 6. System Architecture

### 6.1 아키텍처 개요

시스템은 다음과 같이 4개의 주요 레이어로 구성된다:

1. **Input Layer:** Notion API 연동, 문서 업로드 처리, 텍스트 파싱 (향후 HubSpot API 추가)
2. **Agent Core:** LLM 기반 분석 엔진 (GPT-4o / Claude Sonnet). 정보 구조화, 스코어링, 리스크 분석, 유사 사례 검색 등 수행
3. **Data Layer:** 프로젝트 이력 DB (Vector DB + 관계형 DB), 평가 기준 설정 저장소
4. **Output Layer:** 대시보드 UI, PDF 리포트 생성, Slack/Teams Webhook 알림

### 6.2 기술 스택

| 구분 | 기술 | 비고 |
|---|---|---|
| LLM | OpenAI GPT-4o / Claude Sonnet | 용도별 최적 모델 선택 (A/B 테스트) |
| Backend | Python (FastAPI) | LLM 연동 및 Agent 오케스트레이션 |
| Vector DB | Pinecone / Chroma | 과거 프로젝트 유사도 검색용 |
| RDB | PostgreSQL | 평가 결과, 설정, 사용자 데이터 저장 |
| Frontend | React (Dashboard) | 스코어카드 및 분석 결과 표시 |
| Input 연동 | Notion API | MVP 기준. 향후 HubSpot API 추가 |
| 알림 | Slack Webhook | 분석 완료 알림 및 요약 전송 |

### 6.3 보안 고려사항

- 외부 LLM API 사용 가능 (고객 정보 전송 허용 확인 완료)
- API 통신 시 HTTPS 암호화 적용
- 사내 직원 전용 (10명 이내) 접근 제어 적용
- LLM API Zero Data Retention (ZDR) 옵션 활용 권장

## 7. Output Specification

OathKeeper의 분석 결과는 다음 3가지 채널을 통해 복합적으로 제공된다:

### 7.1 대시보드 스코어카드

- 종합 점수 및 Go/No-Go/조건부 판단 결과 시각적 표시
- 7개 평가 기준별 점수 바 차트 (레이더 차트 또는 바 차트)
- 리스크 요인 요약 및 경고 배지 표시
- 유사 과거 프로젝트 Top 3 비교 요약

### 7.2 PDF 분석 리포트

경영진 보고용 상세 분석 문서로, Deal 개요, 평가 결과, 리스크 분석, 유사 사례 비교, 권고 사항을 포함한 완결된 문서 자동 생성.

### 7.3 Slack/Teams 알림

분석 완료 시 요약 정보와 함께 대시보드 링크를 관련 채널에 자동 전송. 판단 결과(이모지 포함)와 주요 리스크 요약을 포함.

## 8. Development Roadmap

### 8.1 MVP (Phase 1) — 3개월

| 기간 | 주요 작업 | 산출물 | 완료 기준 |
|---|---|---|---|
| 1~2주차 | 평가 기준 확정, 프롬프트 설계, Notion 폼 설계 | 평가 기준 문서, Prompt 초안 | 이해관계자 리뷰 완료 |
| 3~4주차 | Agent Core 개발 (LLM 연동, 스코어링 로직) | 분석 엔진 v1 | 테스트 Deal 5건 분석 성공 |
| 5~6주차 | Notion API 연동, 문서 업로드 처리, 리스크 분석 | Input Pipeline, 리스크 분석 모듈 | End-to-end 파이프라인 동작 |
| 7~8주차 | 대시보드 UI, PDF 리포트, Slack 알림 | 전체 UI 및 Output 모듈 | 사용자 테스트 가능 |
| 9~12주차 | 과거 프로젝트 DB 구축, 유사 사례 검색, 테스트 및 개선 | MVP 전체 완성 | 사내 파일럿 실시 및 피드백 반영 |

### 8.2 향후 확장 계획 (Post-MVP)

- **Phase 2:** HubSpot CRM 연동으로 자동 Deal 정보 수집
- **Phase 3:** 과거 프로젝트 문서 자동 파싱으로 사례 DB 자동 확장
- **Phase 4:** 의사결정 피드백 루프 (Deal 결과 추적 → 평가 모델 개선)
- **Phase 5:** 외부 고객 제공용 SaaS 버전 검토

## 9. Success Criteria & 평가 계획

### 9.1 MVP 성공 기준

- 세일즈가 Deal 정보 입력 후 24시간 이내 분석 결과 확인 가능
- 경영진이 분석 리포트만으로 초기 판단 가능 (추가 정보 요청 최소화)
- 사용자 만족도 설문 4점 이상 (5점 만점)
- 1개월 파일럿 기간 동안 실제 Deal 5건 이상 분석 완료

### 9.2 파일럿 계획

최초 1개월간 실제 Deal 병행 운영을 통해 검증한다. 기존 구두 보고 프로세스와 함께 운영하면서, 에이전트 분석 결과와 실제 의사결정 결과를 비교하여 정확도와 유용성을 평가한다.

## 10. 가정 사항 및 제약 조건

### 10.1 가정 사항

- 세일즈 담당이 Notion 폼을 통해 최소한의 필수 정보를 입력할 의지가 있음
- 경영진이 AI 분석 결과를 의사결정 참고 자료로 활용할 의향이 있음
- 외부 LLM API 사용에 대한 보안 승인이 완료된 상태
- MVP 단계에서는 과거 프로젝트 데이터를 수동으로 입력하는 것을 허용

### 10.2 제약 조건

- MVP 개발 기간 3개월 이내
- 초기 사용자 10명 이내 (소규모 운영)
- 과거 프로젝트 데이터가 산재되어 있어 초기 DB 구축에 수동 작업 필요
- LLM API 비용 예산 관리 필요 (Deal당 API 호출 비용 모니터링)
- 에이전트는 최종 의사결정자가 아닌 **의사결정 지원 도구** 역할 (최종 판단은 경영진)

## Appendix: 용어 정의

| 용어 | 정의 |
|---|---|
| Deal | 고객사로부터 받은 AI 개발 사업 요청 건 |
| Go/No-Go | Deal 수행 여부에 대한 의사결정 (수행 / 비수행 / 조건부 수행) |
| Agent | LLM 기반으로 정보 분석, 판단 지원 등을 수행하는 AI 시스템 |
| Scorecard | 평가 기준별 점수와 종합 판단을 시각적으로 표시한 보고서 |
| Vector DB | 텍스트의 의미적 유사성을 기반으로 검색할 수 있는 데이터베이스 |
| Scope Creep | 프로젝트 진행 중 요구사항이 점진적으로 확대되는 현상 |
